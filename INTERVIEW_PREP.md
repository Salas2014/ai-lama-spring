## 🔄 Как работает RAG (Retrieval-Augmented Generation)

Это главная концепция проекта. Нужно объяснять уверенно.

### Шаг 1 — Ingestion (загрузка знаний)
```
Файл db.txt
    ↓ KnowledgeLoader читает при старте
    ↓ Разбивает на чанки по разделителю "---"
    ↓ GeminiEmbeddingModel превращает каждый чанк в вектор [0.12, 0.87, ...]
    ↓ Qdrant сохраняет вектор + текст + метаданные {source: "db.txt", chunk_index: "0"}
```

### Шаг 2 — RAG Chat (ответ на вопрос)
```
Вопрос пользователя: "как работают индексы?"
    ↓ Gemini превращает вопрос в вектор
    ↓ Qdrant ищет top-3 ближайших векторов (similarity > 0.7)
    ↓ Находит нужные чанки из базы знаний
    ↓ Чанки + вопрос → промпт для Gemini
    ↓ Gemini генерирует ответ ТОЛЬКО на основе найденного контекста
    ↓ Пользователь получает ответ
```

### Зачем RAG а не просто ChatGPT?
- LLM может "галлюцинировать" (выдумывать факты)
- RAG даёт модели конкретный контекст — ответ основан на твоих данных
- Позволяет работать с закрытой/корпоративной информацией

## 🧠 Эмбеддинги — что это и как реализовано

**Эмбеддинг** — это представление текста в виде вектора чисел (768 чисел для text-embedding-004). Похожие по смыслу тексты дают близкие векторы.

### Наша реализация `GeminiEmbeddingModel`:
```java
// Реализует Spring AI интерфейс EmbeddingModel
public class GeminiEmbeddingModel implements EmbeddingModel {

    // Вызывается VectorStore при сохранении документов
    public EmbeddingResponse call(EmbeddingRequest request) { ... }

    // Вызывается при индексации конкретного документа
    public float[] embed(Document document) { ... }

    // Возвращает 768 без вызова API (оптимизация старта)
    public int dimensions() { return 768; }
}
```

**Важно:** `text-embedding-004` доступен только через API `v1` (не `v1beta`). Поэтому создан отдельный `geminiEmbeddingWebClient` с другим base URL.

```yaml
        collection-name: knowledge
        initialize-schema: true
```

## ❓ Вопросы которые могут задать на собеседовании

### Про RAG
**Q: Что такое RAG и зачем он нужен?**
> RAG (Retrieval-Augmented Generation) — паттерн где перед генерацией ответа мы ищем релевантный контекст в базе знаний и передаём его модели. 
> Это нужно чтобы модель отвечала на основе конкретных данных, а не из общих знаний, и не "галлюцинировала".

**Q: Чем RAG отличается от fine-tuning?**
> Fine-tuning — переобучаешь модель на своих данных (дорого, долго, нужно много данных). 
> RAG — подаёшь данные в контекст в рантайме (дёшево, быстро, данные можно обновлять без переобучения).

**Q: Что такое similarity threshold и зачем он нужен?**
> Это порог схожести (у нас 0.7 = 70%). 
> Если найденный чанк похож на запрос менее чем на 70%, он не включается в контекст. Нужен чтобы не передавать в модель нерелевантные данные.

### Про векторные БД
**Q: Что такое векторная БД? Чем Qdrant отличается от PostgreSQL?**

> **PostgreSQL** ищет по точным значениям:
> ```sql
> SELECT * FROM docs WHERE topic = 'индексы'  -- точное совпадение слова
> ```
> Если написать `тема = 'индекс'` или `тема = 'как ускорить поиск'` — ничего не найдёт,
> потому что строки не совпадают побуквенно.
>
> **Векторная БД (Qdrant)** ищет по смыслу:
> ```
> Запрос: "как ускорить поиск в таблице?"
> Найдёт: "индексы ускоряют выборку данных"  ← разные слова, но смысл похож!
> ```
>
> Это работает потому что текст сначала превращается в вектор (массив чисел),
> и похожие по смыслу тексты дают математически близкие векторы.
> Qdrant оптимизирован именно для быстрого поиска таких близких векторов среди миллионов записей.

**Q: Что такое вектор и почему похожие тексты дают близкие векторы?**

> Представь систему координат. Модель обучена так, что слова и фразы
> с похожим смыслом попадают в одну "зону":
>
> ```
>       кот    собака
>        •       •        ← животные (близко друг к другу)

>   индекс  таблица
>      •       •          ← базы данных (близко друг к другу)
>  
>
> Реально это не 2D, а 768 измерений — но принцип тот же.
> "кот" и "собака" математически ближе друг к другу, чем "кот" и "индекс".

**Q: Что такое cosine similarity?**

> Это способ измерить насколько два вектора "смотрят в одном направлении".
>
> Простая аналогия: представь два фонарика направленных из одной точки.
> - Смотрят в одну сторону → угол 0° → cosine = **1.0** (максимальное сходство)
> - Смотрят в разные стороны → угол 90° → cosine = **0.0** (нет сходства)
> - Смотрят в противоположные → угол 180° → cosine = **-1.0**
>
> В нашем проекте порог `0.7` — берём только те чанки,
> где угол между вектором запроса и вектором чанка достаточно мал (тексты похожи на 70%+).

### Про Spring AI
**Q: Что делает интерфейс EmbeddingModel?**
> Это абстракция Spring AI для превращения текста в векторы. 
> VectorStore (Qdrant) использует его автоматически при сохранении и поиске документов. Мы реализовали его через Gemini API.

**Q: Зачем переопределять метод dimensions()?**
> По умолчанию dimensions() вызывает embed("test") чтобы определить размерность. 
> Это лишний API-вызов при каждом старте приложения. М
> ы знаем что text-embedding-004 всегда возвращает 768 — поэтому возвращаем константу.

### Про реактивность
**Q: Зачем WebClient вместо RestTemplate?**
> WebClient — реактивный, неблокирующий HTTP-клиент. 
> Не занимает поток пока ждёт ответа от API. RestTemplate — синхронный, блокирует поток. При большом числе запросов WebClient эффективнее.

**Q: Что такое Mono и Flux?**
> Из Project Reactor. `Mono<T>` — асинхронный контейнер для 0 или 1 значения. 
> `Flux<T>` — асинхронный поток из 0 или N значений. Используем для стриминга ответов чата (SSE).

### Про архитектуру
**Q: Что такое Spring Modulith?**
> Инструмент для модульной архитектуры внутри монолита. 
> Разделяет код на модули (`chat`, `knowledge`) с явными границами. 
> Типы из `.internal` пакетов не доступны другим модулям. Позволяет проверять архитектурные ограничения тестами.

**Q: Почему у тебя два разных base URL для Gemini?**
> `v1beta` — для chat (`generateContent`). `v1` — для embeddings (`embedContent`). Google разнёс разные возможности по разным версиям API. `text-embedding-004` доступен только через стабильный `v1`.

## 🔑 Ключевые решения которые стоит запомнить

1. **Chunking по разделителю `---`** — файлы в `resources/knowledge/` разбиваются на смысловые блоки, а не просто по 2000 символов. Это улучшает качество поиска.

2. **Идемпотентная загрузка** — `KnowledgeLoader` при старте проверяет через `existsBySource()` уже ли загружен файл в Qdrant. Если да — пропускает. Нет дублей.

3. **Переопределение auto-configuration** — вместо стандартных Spring AI бинов используем свои реализации (собственный `GeminiEmbeddingModel` вместо авто-сконфигурированного).

4. **Два режима анализа**:
   - RAG (`/rag`) — ищет по смыслу в базе, быстро, но видит только топ-3 чанка
   - Full analysis (`/topics`) — отправляет весь файл в Gemini, медленнее, но полный анализ
